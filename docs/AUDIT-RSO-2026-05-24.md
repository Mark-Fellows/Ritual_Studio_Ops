# Ritual Studio Ops - Setup Audit

**Audit date:** 2026-05-24
**Scope:** `Ritual_Studio_Ops` project only (the merged TM + CM application)
**Auditor:** Claude (Cowork)
**Method:** Static review of all repository files, plus live verification against the Supabase project (`rfjygyqijwgkmxboddup`) and the Cloudflare account (`db7b2002741f559dc7d8558afc1abf07`) using the Supabase and Cloudflare MCP connectors.

---

## 1. Executive summary

The Ritual Studio Ops (RSO) project is broadly real and functional: the merged application is deployed and live at `https://ritual-studio-ops.pages.dev`, the Phase 1 database tables exist with sensible structure, the cover and Momence service code is syntactically sound (all 54 Python files compile), and the documentation set is unusually thorough. The merger is genuinely most of the way through Phase 5.

However, the audit found **five high-severity issues** that undermine the accuracy and authenticity of the setup, and a further **eight medium-severity issues**. The most consequential are:

- The Phase 1 security deliverable - the `teacher_directory` view that was meant to replace an insecure anonymous-read policy - **does not protect anything**. It is a `SECURITY DEFINER` view and the `anon` role can read it, so unauthenticated callers can still read every teacher's name. The leak that Phase 1 set out to close is still open.
- **12 database tables have Row Level Security disabled**, so the public anonymous key (which is embedded in the deployed page source) can read and write operational and personal data directly.
- The **Phase 5 reconciliation script reports false data** ("0 teachers" against a live count of 79) and prints "Parallel run is GREEN" while two of its four checks are silently blind.
- The **canonical local repository has no Git remote**; the GitHub remote is configured only on a **stale, nested duplicate copy** of the entire project that is sitting inside the project folder.
- The **Phase 1 migration file in the repository is not valid SQL** and does not match what was actually applied to the database.

None of these is visible from a casual look at the project, and several directly contradict the project's own documented design rules and lessons learned. They should be resolved before Phase 6 cutover.

## 2. Scope and method

This audit covers the `Ritual_Studio_Ops` folder only, as requested. The legacy `Ritual_Cover_Management` and `Ritual_Teacher_Management` projects were not separately audited, although the live database is shared with them.

Work performed:

- Full file inventory of the project tree and both Git repositories found within it.
- Read of every governance document, migration, configuration file and the reconciliation script.
- Live Supabase verification: project metadata, table list and row counts, applied migration history and stored migration SQL, Row Level Security policies, security advisors, Edge Functions, and targeted column and grant checks.
- Live Cloudflare verification: account list, Workers list, and a fetch of the deployed Pages URL.
- Execution of all five `test_phase*.py` suites and a compile check of all Python files.

## 4. High-severity findings

### H1 - Row Level Security is disabled on 12 tables

**Finding.** The Supabase security advisor reports `rls_disabled_in_public` (level ERROR) on twelve tables: `cover_requests`, `cover_candidates`, `cover_notifications`, `whatsapp_messages`, `whatsapp_channels`, `whatsapp_monitor_runs`, `discipline_mappings`, `system_config`, `training_courses`, `trainee_enrollments`, `timeslots` and `trainee_bookings`.

**Evidence.** Live `list_tables` and `get_advisors` both confirm `rls_enabled: false` on these tables. The deployed application (`app/ritual-studio-ops-v2.html`, line 529) embeds the anonymous key `SUPABASE_ANON` in plain text, as every Supabase web app does. With RLS disabled, that anonymous key grants full read and write access to every row of these tables.

**Impact.** `whatsapp_messages` and `cover_requests` contain personal data - teacher names, raw message text and request detail. Anyone who views the page source of the public site can extract the anon key and then read or modify these tables directly through the REST API. This directly contradicts the project's own Design Rule 2 ("authenticated by default - no anon read paths in the merged app"): the rule is enforced in the application UI but not at the database, which is the layer that actually matters. The Phase 1 migration explicitly noted this posture and chose not to fix it ("This migration does not change that posture"), so it is a known gap rather than a regression - but it remains unremediated heading into cutover.

**Recommendation.** Design and apply per-table RLS policies, then enable RLS on all twelve tables. Do not simply enable RLS without policies (that blocks all access). Treat `whatsapp_messages` and `cover_requests` as the priority because they hold personal data.

### H2 - The `teacher_directory` view does not protect teacher data

**Finding.** Phase 1 was meant to remove an insecure `anon_read_teacher_names` policy and replace it with a narrow `teacher_directory` view "gated by RLS on `public.teachers` (authenticated users only)" - this is stated verbatim in the migration file and in LESSONS_LEARNED L-CM-07. In practice the view provides no protection.

**Evidence.** The Supabase advisor flags `security_definer_view` (level ERROR) on `public.teacher_directory`. A `SECURITY DEFINER` view runs with the privileges of its creator, not the caller, so it bypasses Row Level Security on the underlying `teachers` table. A grant check confirms the `anon` role holds `SELECT` (and `INSERT`, `UPDATE`, `DELETE`) on `teacher_directory`. The combination means an unauthenticated caller can read `id, first_name, last_name` for all 79 teachers through `/rest/v1/teacher_directory`, exactly the exposure L-CM-07 set out to eliminate.

**Impact.** The headline security deliverable of Phase 1 is ineffective. The migration's own comment asserting that access is "gated by RLS on `public.teachers`" is inaccurate. The write grants to `anon` on the view are an additional risk that should be tested and revoked.

**Recommendation.** Set the view to invoker security: `ALTER VIEW public.teacher_directory SET (security_invoker = on);`. Revoke `anon` privileges on the view. Re-test that an anonymous request returns no rows.

### H3 - The Phase 5 reconciliation script reports false data

**Finding.** `scripts/reconcile.py` is the daily health check that gates the Phase 6 cutover. Its most recent run (`scripts/reconcile_reports/2026-05-23.txt`) reports "Total teacher records: 0" and "Total audit entries (last 24h): 0", then concludes "All checks passed. Parallel run is GREEN." The live database has 79 teachers.

**Evidence.** `reconcile.py` calls `sb_get()` from `services/cover/config.py`. `sb_get()` uses `SUPABASE_KEY`, which resolves to `SUPABASE_SERVICE_KEY or SUPABASE_ANON_KEY or <hardcoded anon key>`. When `.env` has no service key (the `.env.template` ships it blank), the script runs with the anonymous key. The `teachers` and `audit_log` tables both have RLS enabled with authenticated-only read policies, so the anon key returns zero rows. `cover_requests` (70) and `trainee_bookings` (12) read correctly only because those tables have RLS disabled (see H1). `reconcile_state.json` records the same `"teachers": {"count": 0}`.

**Impact.** Check 1 (audit-log origin - the primary detector for "a legacy app is still writing") and Check 4 (teacher-deletion detection) are both blind. The script cannot detect the very conditions it exists to detect, yet it reports GREEN. The two-week parallel run is therefore not actually being verified for teacher integrity or write origin. A real teacher deletion would not be flagged.

**Recommendation.** Run `reconcile.py` with the service-role key (populate `SUPABASE_SERVICE_KEY` in `.env`, or have the script request `service_key=True` explicitly). Re-baseline `reconcile_state.json` afterwards, because the stored snapshot of `0` teachers will otherwise produce a false "increased 0 -> 79" signal on the next run.

### H4 - Version control and deployment linkage is broken

**Finding.** The canonical project repository (the outer `Ritual_Studio_Ops/.git`, at commit `c649a18`) has **no Git remote configured**. The GitHub remote `https://github.com/Mark-Fellows/Ritual_Studio_Ops.git` is configured only on the **nested duplicate** repository at `Ritual_Studio_Ops/Ritual_Studio_Ops/.git`, which is one commit behind (`28d3239`).

**Evidence.** `git -C Ritual_Studio_Ops remote -v` returns nothing; `git -C Ritual_Studio_Ops/Ritual_Studio_Ops remote -v` returns the GitHub origin. `rso_github_push.bat` runs `git -C "%RSO%" push origin master` against the outer repo, which has no `origin` - that push fails. The outer repo's latest commit, "Add app/index.html for Cloudflare Pages Git integration", is the very commit that bat file creates, so the local `add` and `commit` succeeded but the `push` did not.

**Impact.** The latest work has most likely never reached GitHub. If Cloudflare Pages Git integration is wired to that GitHub repository, it builds from a stale tree. There are also three different deployment procedures in play - `wrangler.toml` documents `wrangler pages deploy app/`, `rso_deploy.bat` deploys a temp folder containing only `index.html`, and `rso_github_push.bat` implies Git-integration auto-deploy - and they would not all deploy the same content.

**Recommendation.** Decide on one deployment path. Add the GitHub remote to the canonical (outer) repository, confirm it can push, and remove the nested duplicate (see M1). Confirm whether Cloudflare Pages is connected to GitHub or deployed directly, and document the single chosen method.

### H5 - The Phase 1 migration file is invalid SQL and does not match the database

**Finding.** `migrations/2026-05-merged-v1.sql` uses `CREATE POLICY IF NOT EXISTS` five times. PostgreSQL has no `IF NOT EXISTS` clause for `CREATE POLICY` in any version. The file, as written, cannot be executed - it errors on the first policy statement. The file header nonetheless claims "All objects use IF NOT EXISTS / OR REPLACE", which is impossible for policies.

**Evidence.** The SQL actually applied is recoverable from `supabase_migrations.schema_migrations` for version `20260521105836`. The applied statement uses plain `CREATE POLICY "disciplines_read_authenticated" ...` with no `IF NOT EXISTS`, and also differs from the repository file in its table comments and section headers. The applied version is correct SQL and succeeded; the repository file is a separate, hand-edited variant that diverged.

**Impact.** The migration under version control is not a faithful record of the change and would fail if re-run for disaster recovery or to build a staging environment. The audit trail between repository and database is broken for the project's most important migration.

**Recommendation.** Replace the repository file with the exact SQL that was applied (export it from `supabase_migrations.schema_migrations`). Going forward, apply migrations from the repository file rather than editing them after the fact.

## 5. Medium-severity findings

### M1 - A full duplicate of the project is nested inside the project

`Ritual_Studio_Ops/Ritual_Studio_Ops/` is a complete second copy of the project (95 files, its own `.git`, HEAD one commit behind the outer repo). It is untracked by the outer repository (it appears as `?? Ritual_Studio_Ops/` in `git status`). As noted in H4, it holds the only working GitHub remote. It is almost certainly a `git clone` left in place by accident. It is confusing, doubles the on-disk footprint, and creates genuine ambiguity about which tree is authoritative. **Recommendation:** after migrating the remote to the outer repo (H4), delete the nested copy.

### M2 - `config.py` cannot read the `system_config` table

`services/cover/config.py`, function `get_config_value()`, issues `system_config?key=eq.{key}&select=value`. The table's columns are `config_key` and `config_value`, not `key` and `value` (confirmed live and in `01_cover_schema.sql`). PostgREST returns HTTP 400; the function catches all exceptions and returns the default. The `system_config` table holds 6 rows and exists specifically so thresholds "can be updated without code changes" - but any caller of this function silently ignores it. **Recommendation:** correct the query to `config_key` / `config_value` and verify which stage scripts depend on it.

### M3 - `ritual-studio-ops-v1.html` is truncated

The file ends mid-string (`if(!rows.length){el.innerHTML='<p style`) with no closing `</html>` and no trailing newline. This is the OneDrive write-truncation failure the project itself documented in L-MG-04 and L-MG-05. `v1` is superseded by `v2`, so production is unaffected, but the corrupt file is committed and is served publicly by Cloudflare Pages (the whole `app/` directory is the build output) at `/ritual-studio-ops-v1.html`. **Recommendation:** remove `v1` and the reference copies from `app/`, or move them out of the Pages build directory.

### M4 - `momence_sessions` has RLS enabled but no policy

The advisor reports `rls_enabled_no_policy` on `momence_sessions`. The table holds 7,365 rows, but with RLS enabled and no policy, every read via the anon or authenticated key returns zero rows. Its sibling Phase 1 tables (`momence_members`, `momence_bookings`, `momence_sync_runs`) all received an authenticated read policy; `momence_sessions` did not, because it pre-dated the Phase 1 migration. Any application code that reads `momence_sessions` through a user key receives nothing (service-key callers are unaffected). **Recommendation:** add an authenticated read policy consistent with the other `momence_*` tables.

### M5 - Six live tables are undocumented

The database contains 30 tables; the DOCS_INDEX "tables-and-owners matrix" - which states it is authoritative and "updated each migration" - lists 24. The six missing tables are `membership_types` (187 rows), `class_group_mappings` (87), `locations` (3), `rooms` (5), `location_aliases` (8) and `teacher_pay_rate_tiers` (8). The comments on `membership_types` and `class_group_mappings` reference "RSO Settings -> Reference Data", so they are RSO-adjacent, yet none appear in DOCS_INDEX, the CHANGELOG or the `migrations/` folder, and none are in the Supabase migration history. **Recommendation:** document these tables (or formally exclude them) and reconcile how they were created.

### M6 - A `locations` table duplicates the canonical `studios` table

Design Rule 3 mandates a single source of truth for taxonomy. Phase 1 created `studios` as the canonical studio/location reference (3 rows). A separate `locations` table (3 rows), plus `rooms` and `location_aliases`, also exists, created by a parallel workstream. There are now two competing location references and no documentation of which is canonical. This is precisely the duplication the merger was intended to prevent. **Recommendation:** decide which table is canonical, migrate consumers onto it, and retire the other.

### M7 - Test suites no longer pass as claimed

Running the suites today: `test_phase0.py` prints "99/99 passed" then crashes with `NameError: name 'n' is not defined` (line 218 - the summary block is duplicated and the first copy contains a stray `n(results)` line). `test_phase2.py` fails 2 of 37 checks, because it still asserts that `ritual-studio-ops-v1.html` contains `WRITES_ENABLED = false` and ends with `</html>` - both untrue now (`v1` was write-enabled in Phase 4, and `v1` is truncated, per M3). The CHANGELOG's "99/99" and "37/37" claims were true when written but are not reproducible today. `test_phase5.py` passes 28/28, but it only checks that `reconcile.py` exists and is shaped correctly - it never runs the script against data, which is why it did not catch H3. `test_phase3.py` (65/65) and `test_phase4.py` (24/24) pass as documented. **Recommendation:** fix `test_phase0.py`, update or retire `test_phase2.py`, and add a behavioural check to the Phase 5 suite.

### M8 - The `parse-cover-request` Edge Function has no source in the repository

The function is live (`verify_jwt: true`, status ACTIVE) and the app calls it correctly. The CHANGELOG states it was "deployed directly via Supabase MCP - no local file." There is therefore no version-controlled copy: it cannot be reviewed, diffed or redeployed from the repository, and would be lost if the Supabase project were rebuilt. **Recommendation:** export the function source into the repository (for example under `services/` or a `supabase/functions/` directory) and treat it as version-controlled.

## 6. Low-severity and informational findings

**L1 - Documentation staleness.** DOCS_INDEX section 7 lists `ritual-studio-ops-v1.html` as the active merged app but omits `v2` and `index.html`, although `v2` is what is deployed. The DOCS_INDEX header says "Phase 5 complete" while README says Phase 5 is "In progress". README states "Phase 1 will add ... `momence_sessions` (empty)", but the CHANGELOG correctly records that `momence_sessions` already existed with 7,365 rows. `app/README.md` describes only Phase 0 reference copies and a "Phase 2 target" naming scheme (`ritual-studio-ops-NNN.html`) that the actual `v1`/`v2` files do not follow. The project's own Rule 7 ("update DOCS_INDEX before closing any session") was not honoured for `v2`, `index.html`, the source-column migration or the Edge Function.

**L2 - CHANGELOG gap.** Two migrations recorded in Supabase on 2026-05-19 (`add_classification_override_columns_to_whatsapp_messages`, `add_dual_model_nlp_columns_to_whatsapp_messages`) do not appear in the master CHANGELOG.

**L3 - `index.html` duplicates `v2`.** `app/index.html` is byte-identical to `app/ritual-studio-ops-v2.html`. The two will drift the moment `v2` becomes `v3`, unless re-copied. A redirect stub or a build step would be safer and is more consistent with the "version every file" rule (L-TM-07).

**L4 - Line-ending churn.** The working tree shows 17 files as modified, but the changes are line-ending differences only (the sampled file, `services/momence/config.py`, shows every line removed and re-added with identical text). The repository has no `.gitattributes`, so line endings are unmanaged. This noise masks real changes in `git status`.

**L5 - Reference files served publicly.** Because the entire `app/` folder is the Cloudflare Pages build output, superseded and reference files (`ritual-studio-ops-v1.html`, `ritual-teacher-management31.html`, `cover_dashboard_ref.html`, `teacher_portal_ref.html`, `README.md`) are all reachable on the public site.

**L6 - Role-based access control is app-only.** The advisor flags `teachers` and `class_history` write policies as `USING (true)` - any authenticated user may insert, update or delete any row. The "five-role RBAC" is enforced only in the client; the database grants every signed-in user full write access.

**L7 - Minor hygiene.** `config.py` hardcodes the anon JWT as a fallback (the anon key is public by design, but it should not be in source). `cover_candidates.response` lists `NULL` inside an `IN (...)` CHECK, which is redundant. `reconcile.py` uses the deprecated `datetime.utcnow()`. There are two near-identical trigger functions, `set_updated_at` and `_set_updated_at`. The advisor also notes mutable `search_path` on three functions, an anon-executable `SECURITY DEFINER` function `get_my_role()`, and that leaked-password protection is disabled in Auth. The manual cover-request feature added in `v2` has not yet produced a single row (`cover_requests` is 70/70 `source = whatsapp`), so it is live but unexercised.

## 7. What is working correctly

The audit also confirmed a substantial amount of sound work, which should be stated plainly:

- The merged application is genuinely deployed and live at `https://ritual-studio-ops.pages.dev`, serving the correct build (`v2`, Phase 5).
- Application linkages are intact: the Supabase URL, anon key and the `/functions/v1/parse-cover-request` call all reference the correct project, and `detectSessionInUrl` is correctly set to `true`.
- All 54 Python files across `services/` and `scripts/` compile without error.
- The four Phase 1 tables exist with correct structure, seed data (5 disciplines; 3 studios with Mermaid correctly inactive) and authenticated read policies.
- Both Edge Functions are live; `create-user` correctly has `verify_jwt` disabled, consistent with L-TM-05.
- The Phase 3 cleanup is real: no hardcoded OneDrive Momence paths remain in the cover stage scripts; the remaining `sys.path.insert` calls are the intended relative-import mechanism.
- `01_cover_schema.sql` is well-formed and genuinely idempotent.
- `reconcile.py` is well-structured (graceful degradation, snapshot diffing, meaningful exit codes); its flaw is solely the authentication key it runs under.
- The Cloudflare account in `wrangler.toml` matches the live account exactly.

## 8. Recommended remediation order

1. **H2 and H1** - close the teacher-data exposure: set `teacher_directory` to invoker security and revoke anon grants, then design and enable RLS on the 12 exposed tables. These are live data-protection issues.
2. **H4 and M1** - fix version control: add the remote to the canonical repository, confirm a push succeeds, delete the nested duplicate, and document the single deployment method.
3. **H3** - run `reconcile.py` with the service key and re-baseline the snapshot, so Phase 5 verification is actually meaningful before Phase 6.
4. **H5** - replace the migration file with the SQL that was actually applied.
5. **M2, M4** - fix the `system_config` query and add the `momence_sessions` read policy.
6. **M5, M6** - document or reconcile the six undocumented tables and the `studios`/`locations` duplication.
7. **M3, M7, M8, L1-L7** - housekeeping: remove the corrupt `v1`, repair the test suites, version-control the Edge Function, refresh the documentation and add a `.gitattributes`.

Items 1 to 3 should be completed before Phase 6 cutover.
